Introducción

SQL Injection es una de las vulnerabilidades más relevantes en el ámbito de la seguridad web, ya que permite alterar consultas hacia la base de datos mediante la manipulación de entradas no validadas correctamente. A través de este tipo de ataque, un atacante puede consultar información sensible, omitir mecanismos de autenticación, modificar registros e incluso comprometer por completo la confidencialidad e integridad de la aplicación. Por esta razón, SQL Injection continúa siendo un tema fundamental dentro de las pruebas de penetración y del desarrollo seguro de software.

El presente documento técnico-profesional tiene como finalidad documentar de manera estructurada la resolución de los laboratorios oficiales de SQL Injection de la plataforma PortSwigger Web Security Academy, empleando Burp Suite como herramienta principal para la interceptación, modificación y análisis de peticiones HTTP. A lo largo del desarrollo se presenta el fundamento teórico del ataque, la clasificación de sus variantes, el procedimiento práctico seguido en cada laboratorio, la explicación de los payloads utilizados, así como las medidas de mitigación recomendadas para prevenir este tipo de vulnerabilidades en aplicaciones web.

Asimismo, este trabajo busca demostrar no solo la ejecución práctica de los laboratorios, sino también la comprensión técnica de la lógica detrás de cada explotación, el impacto de seguridad que representa una inyección SQL y la importancia de aplicar controles defensivos como consultas parametrizadas, validación de entradas y principios de mínimo privilegio. De esta manera, el documento integra tanto el enfoque ofensivo de identificación y explotación, como la visión defensiva orientada a la protección de sistemas y datos.

Marco Teórico — Funcionamiento de bases de datos relacionales

Las bases de datos relacionales son sistemas diseñados para almacenar, organizar y consultar información de manera estructurada mediante tablas. Cada tabla se compone de filas y columnas: las filas representan registros individuales, mientras que las columnas representan los atributos o campos de dichos registros. Por ejemplo, en una aplicación web puede existir una tabla de usuarios con columnas como id, username, password y email, donde cada fila corresponde a un usuario distinto.

El modelo relacional permite que diferentes tablas se vinculen entre sí mediante claves primarias y claves foráneas. Una clave primaria identifica de forma única cada registro dentro de una tabla, mientras que una clave foránea establece relaciones con registros de otra tabla. Gracias a esto, las aplicaciones pueden consultar datos relacionados sin necesidad de duplicar información. Por ejemplo, una tienda en línea puede tener una tabla de productos y otra de categorías, relacionándolas para mostrar los artículos de una categoría específica.

Para interactuar con una base de datos relacional se utiliza el lenguaje SQL (Structured Query Language), el cual permite realizar operaciones como consultar, insertar, actualizar y eliminar datos. Cuando una aplicación web recibe información del usuario, como credenciales de inicio de sesión o filtros de búsqueda, normalmente construye consultas SQL para recuperar la información correspondiente desde la base de datos. Si esta interacción no se implementa de forma segura, un atacante puede manipular la consulta original e introducir instrucciones maliciosas, dando lugar a vulnerabilidades como SQL Injection.

La importancia de comprender el funcionamiento de las bases de datos relacionales dentro de la ciberseguridad radica en que la mayoría de las aplicaciones web modernas dependen de este modelo para gestionar información sensible. Por ello, conocer cómo se estructuran las tablas, cómo se relacionan y cómo se ejecutan las consultas SQL es fundamental para entender tanto el origen como la explotación de vulnerabilidades en el acceso a los datos.

Marco Teórico — Consultas SQL básicas (SELECT, WHERE, UNION, ETC.)

SQL, por sus siglas en inglés Structured Query Language, es el lenguaje utilizado para interactuar con bases de datos relacionales. Mediante SQL es posible consultar, insertar, actualizar y eliminar información almacenada en tablas. En el contexto de aplicaciones web, las consultas SQL se ejecutan constantemente para autenticar usuarios, mostrar productos, filtrar categorías o recuperar datos específicos según la solicitud realizada por el cliente.

SELECT

Una de las instrucciones más utilizadas es SELECT, la cual permite recuperar información de una o varias columnas de una tabla. Por ejemplo, una consulta como SELECT username FROM users solicita los valores de la columna username de la tabla users. Si se desea obtener todos los campos de una tabla, puede emplearse SELECT * FROM users, aunque en entornos reales se recomienda seleccionar únicamente las columnas necesarias para reducir exposición de datos y mejorar el rendimiento.

WHERE

Otra cláusula fundamental es WHERE, utilizada para establecer condiciones dentro de una consulta. Esta cláusula permite filtrar resultados con base en criterios específicos. Por ejemplo, SELECT * FROM users WHERE username = 'administrator' busca únicamente el registro cuyo nombre de usuario sea administrator. En aplicaciones web, esta lógica suele emplearse en formularios de inicio de sesión, búsquedas internas o filtros por categoría. Precisamente por ello, cuando los valores que llegan a la cláusula WHERE no son validados correctamente, pueden convertirse en un punto crítico para ataques de inyección SQL.

UNION

La instrucción UNION permite combinar los resultados de dos consultas SQL en un solo conjunto de resultados. Para que funcione correctamente, ambas consultas deben devolver el mismo número de columnas y, además, estas deben ser compatibles en tipo de dato y orden. Por ejemplo, un atacante puede aprovechar una vulnerabilidad para inyectar una consulta con UNION SELECT y así recuperar información de otras tablas, como nombres de usuarios o contraseñas. Esta técnica fue precisamente una de las más utilizadas en los laboratorios desarrollados, ya que permite exfiltrar información desde la misma respuesta de la aplicación.

Comprender el uso de SELECT, WHERE y UNION es esencial para analizar cómo una aplicación consulta datos y cómo un atacante puede alterar ese comportamiento cuando existe una implementación insegura. Estas instrucciones constituyen la base de muchas explotaciones de SQL Injection, por lo que su estudio resulta indispensable tanto para el análisis ofensivo como para la aplicación de mecanismos de defensa en seguridad informática.

Marco Teórico — ¿Qué es SQL Injection?

SQL Injection es una vulnerabilidad de seguridad que ocurre cuando una aplicación web inserta directamente datos proporcionados por el usuario dentro de una consulta SQL sin aplicar mecanismos adecuados de validación o sanitización. Como consecuencia, un atacante puede modificar la estructura de la consulta original y ejecutar instrucciones no previstas por el desarrollador. Esto permite alterar la lógica de autenticación, acceder a información confidencial, enumerar tablas y columnas, o incluso comprometer por completo la base de datos.

Este tipo de ataque aprovecha la confianza indebida que la aplicación deposita en la entrada del usuario. Por ejemplo, si un sistema concatena directamente un valor ingresado en un formulario de inicio de sesión o en un filtro de categoría, el atacante puede introducir caracteres especiales, operadores lógicos o sentencias SQL adicionales. De esta manera, la base de datos interpreta la entrada maliciosa como parte legítima de la consulta, en lugar de tratarla únicamente como un dato.

La gravedad de SQL Injection radica en que puede afectar directamente los tres pilares de la seguridad de la información. En términos de confidencialidad, permite acceder a datos sensibles como usuarios, contraseñas o información personal. En cuanto a la integridad, puede facilitar la modificación o eliminación de registros. Respecto a la disponibilidad, un atacante puede provocar errores, saturación del sistema o incluso dañar la operación de la base de datos. Por ello, SQL Injection ha sido históricamente considerada una de las vulnerabilidades más críticas en entornos web.

Dentro de las pruebas de penetración, SQL Injection es una técnica fundamental porque demuestra cómo una falla aparentemente simple en la validación de entradas puede derivar en un compromiso total del sistema. Además, sigue siendo una vulnerabilidad relevante en marcos de referencia reconocidos como OWASP Top 10, debido a su impacto y frecuencia en aplicaciones inseguras. Por esta razón, comprender su funcionamiento no solo permite explotarla en un entorno controlado de laboratorio, sino también diseñar mejores estrategias de prevención y endurecimiento de aplicaciones reales.

Marco Teórico — Clasificación de ataques SQLi

Los ataques de SQL Injection pueden clasificarse de acuerdo con la forma en que el atacante introduce la consulta maliciosa y obtiene la información del servidor. Esta clasificación permite comprender mejor el comportamiento de cada técnica, su nivel de complejidad y el tipo de evidencia que puede observarse durante una prueba de seguridad. En términos generales, las variantes más conocidas son In-band, Blind y Out-of-band.

In-band (Union-based, Error-based)

La inyección SQL de tipo In-band ocurre cuando el atacante utiliza el mismo canal de comunicación para enviar la carga maliciosa y recibir la respuesta del servidor. Es decir, la explotación y la extracción de datos se realizan directamente a través de la respuesta HTTP de la aplicación. Esta es una de las formas más comunes y fáciles de identificar dentro de laboratorios y entornos vulnerables, ya que los resultados suelen mostrarse de forma inmediata en la página.

Dentro de esta categoría se encuentran dos subtipos principales:

  • Union-based SQL Injection. Se basa en el uso del operador UNION para combinar la consulta original de la aplicación con una consulta inyectada por el atacante. Su propósito es mostrar en la misma respuesta datos de otras tablas, como nombres de usuario, contraseñas o versiones del motor de base de datos. Para que funcione correctamente, ambas consultas deben devolver el mismo número de columnas y tipos de datos compatibles.
  • Error-based SQL Injection. En esta técnica, el atacante provoca errores intencionales en la base de datos para que el sistema revele información útil en los mensajes de error. Si la aplicación muestra detalles internos del motor SQL, dichos errores pueden exponer nombres de tablas, columnas, estructura de la consulta o incluso datos sensibles. Esta técnica depende de una configuración insegura que permita mostrar errores detallados al usuario.

Blind (Boolean-based, Time-based)

La inyección SQL de tipo Blind se presenta cuando la aplicación no devuelve mensajes de error visibles ni muestra directamente los resultados de la consulta maliciosa. En este escenario, el atacante debe inferir la información observando cambios indirectos en la respuesta del sistema, como variaciones en el contenido, el comportamiento o el tiempo de respuesta. Aunque este tipo de explotación suele ser más lenta, sigue siendo altamente peligrosa.

Sus dos variantes más comunes son:

  • Boolean-based Blind SQL Injection. Consiste en enviar condiciones lógicas que provocan respuestas diferentes según si la expresión es verdadera o falsa. Por ejemplo, la página puede mostrar un mensaje como "Welcome back" cuando la condición se cumple, y omitirlo cuando no se cumple. A partir de estas diferencias sutiles, el atacante puede extraer información carácter por carácter.
  • Time-based Blind SQL Injection. En esta variante, el atacante introduce funciones que obligan a la base de datos a retrasar su respuesta durante un número específico de segundos. Si la respuesta tarda exactamente el tiempo esperado, se confirma que la condición inyectada fue verdadera. De esta forma, el tiempo de respuesta se convierte en el canal para inferir datos aun cuando la página no muestre resultados visibles.

Out-of-band

La inyección SQL Out-of-band ocurre cuando la extracción de información no se realiza a través de la misma respuesta HTTP de la aplicación, sino por un canal externo controlado por el atacante, como DNS o HTTP. Este tipo de ataque suele utilizarse cuando no es posible obtener datos mediante respuestas directas ni mediante diferencias visibles o temporales.

Aunque es menos común que las variantes anteriores, puede ser muy efectiva en escenarios donde el servidor tiene conectividad hacia servicios externos. Por ejemplo, una carga maliciosa puede forzar a la base de datos a realizar una consulta DNS hacia un dominio controlado por el atacante, enviando en esa petición información sensible codificada. Debido a ello, este tipo de SQL Injection representa un riesgo importante en infraestructuras mal segmentadas o con configuraciones permisivas.

Comprender la clasificación de los ataques SQL Injection es fundamental para interpretar correctamente el comportamiento de cada laboratorio, seleccionar la técnica de explotación adecuada y proponer medidas defensivas acordes al tipo de vulnerabilidad identificada. Cada variante demuestra que una mala validación de entradas puede convertirse en una puerta de acceso crítica hacia la base de datos y, en consecuencia, hacia la información sensible de una aplicación web.

Marco Teórico — Impacto en la confidencialidad, integridad y disponibilidad

La vulnerabilidad de SQL Injection puede afectar directamente los tres principios fundamentales de la seguridad de la información: confidencialidad, integridad y disponibilidad. Esto se debe a que la explotación de una inyección SQL permite interactuar de forma no autorizada con la base de datos, que normalmente contiene información crítica para el funcionamiento de la aplicación.

Confidencialidad

En cuanto a la confidencialidad, el impacto ocurre cuando un atacante logra consultar datos que no debería conocer, como credenciales de acceso, información personal de usuarios, correos electrónicos, historiales, registros internos o datos administrativos. En varios laboratorios de esta práctica fue posible extraer nombres de usuario y contraseñas, lo que demuestra cómo una mala validación de entradas puede terminar exponiendo información sensible.

Integridad

Respecto a la integridad, SQL Injection también puede permitir la alteración de la información almacenada. Si la cuenta de base de datos utilizada por la aplicación tiene privilegios suficientes, un atacante podría modificar registros, cambiar contraseñas, insertar datos falsos o eliminar información existente. Esto compromete la confiabilidad de los datos y afecta directamente la operación del sistema.

Disponibilidad

En términos de disponibilidad, una inyección SQL puede provocar fallos en la aplicación, errores internos del servidor, bloqueos de procesos o consultas costosas que consuman recursos excesivos. Incluso, en escenarios más graves, podría utilizarse para borrar tablas o afectar la estructura de la base de datos, dejando el servicio fuera de operación. Por ello, SQL Injection no solo representa un problema de exposición de información, sino un riesgo integral para todo el sistema.

Marco Teórico — Relación con OWASP Top 10

La vulnerabilidad de SQL Injection se encuentra estrechamente relacionada con los riesgos más críticos identificados por OWASP Top 10, una de las referencias más importantes en seguridad de aplicaciones web. OWASP, cuyas siglas corresponden a Open Worldwide Application Security Project, mantiene esta clasificación con el propósito de concientizar sobre las fallas más comunes y peligrosas que afectan a los sistemas modernos. Dentro de este marco, SQL Injection ha sido históricamente una de las vulnerabilidades más representativas.

En versiones anteriores del OWASP Top 10, los ataques de inyección aparecían de forma explícita como una categoría principal debido a su alta frecuencia e impacto. Aunque en versiones más recientes la clasificación se ha ampliado y reorganizado, SQL Injection continúa formando parte del grupo de vulnerabilidades relacionadas con fallas de validación, interpretación insegura de entradas y ejecución no intencionada de comandos sobre sistemas internos. Esto demuestra que, a pesar de los avances en frameworks y buenas prácticas de desarrollo, la inyección sigue siendo una amenaza vigente.

La razón por la que SQL Injection ocupa un lugar tan importante dentro de OWASP es que una sola vulnerabilidad de este tipo puede escalar rápidamente a una afectación completa del sistema. A diferencia de otras fallas que requieren múltiples condiciones para ser explotadas, una inyección SQL exitosa puede bastar para obtener acceso a información confidencial, evadir autenticaciones, comprometer cuentas administrativas o incluso tomar control de la lógica de la aplicación. Por ello, OWASP la considera una falla de alto riesgo tanto por su explotabilidad como por su impacto potencial.

Relacionar SQL Injection con OWASP Top 10 es importante dentro de este proyecto porque permite ubicar los laboratorios realizados dentro de un contexto profesional y reconocido internacionalmente. No se trata solo de resolver ejercicios técnicos, sino de comprender que este tipo de prácticas reproducen vulnerabilidades reales presentes en sistemas productivos. Por esta razón, el análisis de SQL Injection y sus mecanismos de defensa constituye una competencia esencial para cualquier estudiante o profesional orientado a la seguridad informática, auditoría de aplicaciones y pruebas de penetración.

Marco Teórico — Importancia en pruebas de penetración

SQL Injection es una de las vulnerabilidades más importantes dentro de las pruebas de penetración, ya que permite evaluar de forma directa la seguridad en la interacción entre la aplicación web y la base de datos. Su detección demuestra que existe una validación deficiente de entradas y que el sistema podría ser manipulado para ejecutar instrucciones no autorizadas.

Durante un pentest, identificar una vulnerabilidad de SQL Injection tiene un alto valor porque puede conducir rápidamente a resultados críticos, como evasión de autenticación, enumeración de tablas y columnas, recuperación de credenciales, acceso a cuentas privilegiadas o extracción de información sensible. Por esta razón, suele considerarse una de las pruebas más relevantes dentro de la auditoría de aplicaciones web.

Además, la explotación controlada de SQL Injection permite medir el impacto real de la falla y generar recomendaciones concretas de remediación. No se trata únicamente de demostrar que la vulnerabilidad existe, sino de analizar hasta dónde puede llegar un atacante si la organización no implementa consultas parametrizadas, validación de entradas, controles de privilegios y manejo seguro de errores.

Su importancia en pruebas de penetración también radica en que combina análisis técnico, comprensión de bases de datos, manipulación de peticiones HTTP y uso de herramientas especializadas como Burp Suite. Por ello, el estudio y documentación de estos laboratorios fortalece habilidades esenciales para la evaluación ofensiva y defensiva de aplicaciones web.

Marco Teórico — Tipos de payloads

En SQL Injection, un payload es la cadena de texto o instrucción que el atacante inserta en un parámetro vulnerable con el objetivo de alterar el comportamiento de la consulta SQL original. Los payloads varían según el tipo de inyección, la base de datos utilizada y el objetivo del ataque, ya sea autenticar sin credenciales, identificar columnas, extraer datos o provocar retrasos en la respuesta.

Payloads de evasión de autenticación

Un tipo común son los payloads de evasión de autenticación, utilizados para modificar la lógica de una consulta de inicio de sesión. Su propósito es hacer que la condición del WHERE se evalúe como verdadera sin necesidad de conocer la contraseña legítima. Este tipo de payload demuestra cómo una autenticación insegura puede ser anulada con una entrada manipulada.

Payloads UNION-based

También existen los payloads UNION-based, que se emplean para combinar la consulta original de la aplicación con una consulta inyectada por el atacante. Estos payloads permiten recuperar información de otras tablas, como usuarios, contraseñas, nombres de columnas o versiones del motor de base de datos. Suelen incluir valores NULL para identificar el número de columnas y después reemplazarlos por texto o datos de interés.

Payloads Error-based

Otro grupo importante son los payloads error-based, diseñados para provocar errores controlados en la base de datos y obtener información a partir de los mensajes devueltos por el servidor. Cuando la aplicación expone errores detallados, estos payloads pueden revelar datos sensibles, estructura de tablas o resultados de subconsultas.

Payloads Blind (Boolean y Time-based)

En el caso de Blind SQL Injection, se utilizan payloads booleanos y payloads basados en tiempo. Los booleanos introducen condiciones verdaderas o falsas para observar diferencias en la respuesta de la página, mientras que los basados en tiempo emplean funciones de retraso para confirmar si una condición fue ejecutada correctamente por la base de datos. Aunque estos métodos suelen ser más lentos, son muy efectivos cuando la aplicación no muestra errores ni resultados visibles.

Payloads de enumeración y exfiltración

Finalmente, también pueden existir payloads de enumeración y payloads de exfiltración, orientados a descubrir tablas, columnas, tipos de datos o extraer múltiples valores en una sola columna. En todos los casos, el payload representa la parte central del ataque, ya que es el medio por el cual se manipula la consulta original. Por ello, entender su estructura y propósito es indispensable tanto para la explotación en entornos controlados como para la implementación de defensas efectivas.

Marco Teórico — Principios de defensa

La prevención de SQL Injection requiere aplicar controles de seguridad desde el diseño y desarrollo de la aplicación, no únicamente después de detectar la vulnerabilidad. La medida más importante y efectiva consiste en utilizar consultas parametrizadas o sentencias preparadas, ya que estas separan de forma estricta la estructura del comando SQL y los datos introducidos por el usuario. De esta manera, aunque un atacante intente enviar caracteres especiales, operadores lógicos o sentencias maliciosas, la base de datos los interpretará únicamente como datos literales y no como instrucciones ejecutables.

Validación estricta de entradas

Otro principio fundamental es la validación estricta de entradas. Toda información recibida desde formularios, parámetros de URL, encabezados HTTP o cookies debe comprobarse antes de ser procesada. Esto implica verificar longitud, formato, tipo de dato y valores permitidos. Cuando es posible, se recomienda aplicar esquemas de lista blanca, aceptando únicamente valores previamente definidos por la lógica de la aplicación. Aunque la validación por sí sola no sustituye a las consultas parametrizadas, sí contribuye a reducir la superficie de ataque.

Uso de ORMs o frameworks seguros

También es recomendable utilizar ORMs o frameworks seguros, ya que muchos de ellos incorporan mecanismos internos para manejar consultas parametrizadas y abstraer el acceso directo a la base de datos. Sin embargo, su uso no garantiza seguridad automática si el desarrollador implementa consultas dinámicas inseguras o concatena manualmente valores sin control. Por ello, el framework debe entenderse como un apoyo y no como un sustituto de las buenas prácticas de desarrollo seguro.

Principio de mínimo privilegio

El principio de mínimo privilegio en la base de datos es otra medida crítica. La cuenta utilizada por la aplicación web no debería tener permisos innecesarios sobre todas las tablas, vistas o procedimientos. Si un atacante logra explotar una inyección SQL, el alcance del daño dependerá en gran parte de los privilegios asociados a esa cuenta. Limitar permisos de lectura, escritura, modificación y administración reduce significativamente el impacto potencial de una explotación.

Monitoreo y manejo seguro de errores

Finalmente, es importante complementar estas defensas con manejo seguro de errores, monitoreo de actividad sospechosa y pruebas periódicas de seguridad. Los mensajes de error no deben exponer detalles internos del motor de base de datos, ya que pueden facilitar ataques de tipo error-based. Asimismo, el monitoreo de consultas anómalas y la realización de auditorías permiten identificar comportamientos sospechosos antes de que deriven en un incidente mayor. En conjunto, estas medidas forman una estrategia defensiva integral contra una de las vulnerabilidades más peligrosas en aplicaciones web.

Tipo de SQLi

SQL Injection In-band — modificación de cláusula WHERE.

Objetivo

Explotar la vulnerabilidad para recibir todos los documentos, incluyendo los no publicados.

Vulnerabilidad

La aplicación no filtra el parámetro category antes de incluirlo en la consulta SQL, permitiendo alterar la lógica original y omitir el filtro restante.

Procedimiento

  1. Iniciar el laboratorio y navegar a cualquier categoría. La URL muestra: /filter?category=Pets
  2. Insertar directamente en la URL una categoría malformada para sobreescribir la query original e ignorar el filtro restante:
/filter?category=Pets' OR 1=1--

Payload

Pets' OR 1=1--

Explicación

  • ' — cierra la cadena de texto de la consulta original.
  • OR 1=1 — condición siempre verdadera que devuelve todos los registros.
  • -- — comenta el resto de la consulta original para evitar errores de sintaxis.

Mitigación

Implementar consultas parametrizadas o sentencias preparadas. Los marcadores de posición separan la lógica SQL de los datos del usuario.

  4. Seleccionar el payload → clic derecho → Extensions → Hackvertor → Encode → Hex entities. Esto convierte los caracteres a entidades XML hexadecimales.
  5. Enviar con el payload codificado → el WAF no detecta el ataque y lo procesa.
  6. Determinar número de columnas: UNION SELECT NULL → 1 columna en la consulta original.
  7. Concatenar usuario y contraseña en esa única columna: 
    UNION SELECT username||':'||password FROM users
  8. El servidor devuelve las credenciales. Copiar la del administrator e iniciar sesión.

Payload

UNION SELECT username||':'||password FROM users

(codificado en entidades XML hexadecimales con Hackvertor antes de enviar)

Explicación

  • UNION SELECT — combina la consulta original con la inyectada.
  • username||':'||password — concatena usuario y contraseña en la única columna disponible.
  • La codificación XML hexadecimal evade el WAF porque este solo analiza texto plano; el servidor decodifica las entidades antes de ejecutar la SQL.

Mitigación

Consultas parametrizadas (defensa primaria). Actualización constante del WAF para detectar técnicas de evasión por codificación. Validación y sanitización de entradas antes del WAF. Principio de mínimo privilegio en la cuenta de BD.

Lab Solved

Conclusión

Los 18 laboratorios realizados demostraron cómo una vulnerabilidad de SQL Injection puede manifestarse de formas muy diversas: desde la simple modificación de una cláusula WHERE hasta la exfiltración de datos por canal DNS completamente fuera de banda. A lo largo de la práctica fue posible identificar que la causa raíz siempre es la misma: la inserción directa de datos del usuario en consultas SQL sin validación ni parametrización.

El uso de Burp Suite como herramienta principal permitió interceptar, analizar y modificar peticiones HTTP de forma controlada, aplicando payloads progresivamente más complejos según el tipo de vulnerabilidad. Los laboratorios de tipo Blind evidenciaron que incluso cuando la aplicación no muestra resultados directos, el comportamiento del sistema (respuestas condicionales, tiempos de respuesta, interacciones DNS) puede convertirse en un canal de extracción de información igualmente efectivo.

La práctica confirma que SQL Injection sigue siendo una amenaza vigente y que la única defensa verdaderamente efectiva es el uso de consultas parametrizadas, complementado con el principio de mínimo privilegio, manejo seguro de errores y monitoreo continuo de consultas anómalas.
